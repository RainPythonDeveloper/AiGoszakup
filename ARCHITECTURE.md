# Архитектура AI-Агента Анализа Госзакупок РК

## Обзор

Архитектура представляет собой **6-слойную вертикальную пайплайн-систему** с потоком данных сверху вниз (от API до пользователя) и обратными запросами (от AI-агента к хранилищу). Ключевой принцип: **LLM не считает сама** — все вычисления делегируются в SQL/Python, а LLM только планирует запросы и формирует человекочитаемые ответы.

## Слой 0: Внешний источник данных

**Компонент:** `goszakup.gov.kz — API OWS v3 (REST + GraphQL)`

### Что делает

Единственный источник истины. API предоставляет доступ к 10+ сущностям: объявления (`/v3/trd-buy`), лоты (`/v3/lots`), договоры (`/v3/contract`), планы (`/v3/plans`), участники (`/v3/subject`), заявки (`/v3/trd-app`), платежи (`/v3/payment`), акты (`/v3/contract-act`), справочники и журнал изменений (`/v3/journal`).

### Почему так

- API поддерживает **REST и GraphQL** — REST для массовой выгрузки (пагинация через `search_after`), GraphQL для сложных запросов с фильтрацией по диапазонам дат и подстановочными символами
- Эндпоинт **`/v3/journal`** критически важен — он возвращает список изменённых объектов за период, что позволяет реализовать инкрементальную синхронизацию (требование ТЗ: задержка <= 24 часа)
- Авторизация через Bearer-токен (срок жизни 1 год)

### Выбранные эндпоинты и обоснование

| Эндпоинт | Сущность | Зачем нужна |
|----------|----------|-------------|
| `/v3/trd-buy` + `/v3/trd-buy/all` | Объявления | Основа анализа — суммы, способы закупки, даты, статусы |
| `/v3/lots` + `/v3/lots/bin/{bin}` | Лоты | Детализация по единицам ТРУ, коды ENSTРУ, цены за единицу |
| `/v3/contract` + `/v3/contract/full` | Договоры | Факт заключения, суммы, стороны, для анализа Fair Price |
| `/v3/contract/subject` | Предметы договора | Конкретные товары/услуги с количествами и ценами |
| `/v3/plans` + `/v3/plans/all` | Годовые планы | Плановые объёмы vs факт — для выявления завышения количеств |
| `/v3/plans/kato` | Места поставки | Региональные коэффициенты для Fair Price |
| `/v3/plans/spec` | Спецификации | Детализация планов по программам |
| `/v3/subject/biin/{bin}` | Заказчики/Поставщики | Информация об организациях, БИН, адреса |
| `/v3/trd-app` | Заявки | Ценовые предложения поставщиков — для оценки конкуренции |
| `/v3/payment/{contract_id}` | Платежи | Казначейские данные — факт оплаты |
| `/v3/contract-act` | Акты | Подтверждение исполнения |
| `/v3/journal` | Журнал изменений | Инкрементальная синхронизация |
| Справочники (КАТО, МКЕЙ, статусы, способы закупки) | Справочники | Нормализация и классификация данных |

## Слой 1: Data Ingestion Layer

**Компоненты:** `REST/GraphQL Connector` | `Journal Sync` | `Rate Limiter`

### Что делает

Забирает данные из API и складывает в Raw-зону хранилища. Работает в двух режимах:

1. **Full Load (первичная загрузка)** — полная выгрузка 3 лет данных (2024–2026) по указанным БИНам через REST с пагинацией (`page=next&search_after=<last_id>`)
2. **Incremental Sync (ежедневная)** — через `/v3/journal?date_from=...&date_to=...` получаем список изменённых объектов (действия U/D), затем точечно перезапрашиваем только их

### Почему так

- **Journal-based инкремент** — единственный надёжный способ гарантировать задержку <= 24h (требование ТЗ). Альтернатива (полная перевыгрузка каждый день) неприемлема по объёмам и нагрузке на API
- **Rate Limiter** — API goszakup.gov.kz имеет лимиты на количество запросов. Без rate limiting токен может быть заблокирован. Реализуется через token bucket algorithm с retry и exponential backoff
- **REST для bulk, GraphQL для точечных запросов** — REST-пагинация через `search_after` эффективнее для полных выгрузок; GraphQL используется для сложных фильтрованных запросов (например, поиск по диапазону дат и сумм)

### Технологии

- Python + `httpx`/`aiohttp` для асинхронных запросов
- Raw JSON сохраняется в staging-таблицы PostgreSQL (тип `JSONB`) для аудита и возможности re-processing

## Слой 2: ETL Pipeline

**Компоненты:** `Cleaning` | `Normalization` | `Enrichment` | `Quality Gate`

### Что делает

**Cleaning (Очистка):**

- Удаление мусора из названий лотов (HTML-теги, лишние пробелы, спецсимволы)
- Стандартизация единиц измерения через справочник МКЕЙ
- Обработка NULL-значений и дубликатов
- Дедупликация записей (один лот может прийти через несколько эндпоинтов)

**Normalization (Нормализация):**

- Приведение БИН/ИИН к единому 12-значному формату
- Парсинг и стандартизация дат (API возвращает разные форматы)
- Нормализация сумм (убираем строковые суффиксы, приводим к decimal)
- Маппинг кодов КАТО на регионы (для региональных коэффициентов Fair Price)
- Приведение кодов ЕНС ТРУ к единой иерархии

**Enrichment (Обогащение):**

- Добавление расчётных полей: цена за единицу = сумма / количество
- Присвоение категорий верхнего уровня по коду ЕНС ТРУ
- Связывание лотов с договорами и планами через FK
- Вычисление индексов инфляции (ИПЦ по данным stat.gov.kz) для временной корректировки цен

**Quality Gate (Контроль качества):**

- Проверка referential integrity (каждый лот ссылается на существующее объявление)
- Валидация диапазонов (суммы > 0, даты в пределах 2024–2026)
- Детекция аномальных строк (суммы = 0 при непустом количестве)
- Логирование отбракованных записей с причинами в таблицу `data_quality_log`

### Почему так

- **Очистка названий лотов** — прямое требование ТЗ. Без неё LLM получит мусорный контекст и снизится точность (требование >= 85%)
- **Quality Gate как отдельный этап** — обеспечивает NFR "контроль качества данных". Позволяет мониторить % отбракованных записей и срабатывать по алерту
- **Enrichment с ИПЦ** — необходим для метрики Fair Price (временной фактор — учёт инфляции)

### Технологии

- Python (pandas/polars для трансформаций)
- Оркестрация через **Apache Airflow** — DAG с задачами: `ingest -> clean -> normalize -> enrich -> validate -> load`

## Слой 3: PostgreSQL Storage

**Компоненты:** `Operational Tables` | `Materialized Views` | `Справочники`

### Почему PostgreSQL, а не ClickHouse

- **PostgreSQL** выбран как основное хранилище по следующим причинам:
  1. Данные реляционные с множеством FK-связей (лот -> объявление -> заказчик -> договор -> поставщик) — реляционная БД лучше подходит
  2. AI-агент генерирует произвольные SQL-запросы — PostgreSQL имеет богатейший SQL-диалект (window functions, CTEs, lateral joins)
  3. Объём данных (3 года по ограниченному набору БИНов) — десятки тысяч — миллионы записей, не Big Data. PostgreSQL справляется
  4. On-prem deployment (требование ТЗ) — PostgreSQL проще развернуть в Docker/K8s, чем ClickHouse
  5. JSONB-поддержка для raw data и гибких полей

- **ClickHouse** был бы оправдан при >100M строк или если основной паттерн — агрегация по времени. Здесь это не так.

### Materialized Views (Витрины данных)

```sql
-- mv_price_statistics: Средневзвешенные цены по ENSTRU + регион + период
CREATE MATERIALIZED VIEW mv_price_statistics AS
SELECT
    enstru_code,
    delivery_kato,
    date_trunc('quarter', sign_date) AS period,
    COUNT(*) AS sample_size,
    SUM(price_per_unit * quantity) / SUM(quantity) AS weighted_avg_price,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY price_per_unit) AS median_price,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY price_per_unit) AS q1,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY price_per_unit) AS q3,
    STDDEV(price_per_unit) AS std_dev
FROM contract_subjects cs
JOIN contracts c ON cs.contract_id = c.id
GROUP BY enstru_code, delivery_kato, date_trunc('quarter', sign_date);

-- mv_volume_trends: Объёмы закупок по ТРУ по годам (для выявления завышений)
-- mv_supplier_stats: Статистика поставщиков (количество контрактов, суммы)
-- mv_anomaly_candidates: Предварительно отфильтрованные подозрительные закупки
```

### Почему именно так

- **Materialized Views** — ТЗ требует формировать ответы с цифрами (средневзвешенные, медианы, отклонения). Предрасчитанные агрегаты ускоряют ответ агента в 10-100x по сравнению с on-the-fly расчётами
- **`price_per_unit` как расчётное поле** — ключевой показатель для всех типов анализа (аномалии, справедливость, объёмы)
- **`delivery_kato`** на уровне лотов и предметов договора — необходим для регионального коэффициента Fair Price
- **`sync_journal`** — отслеживает какие объекты уже обработаны из журнала изменений, предотвращает повторную обработку

## Слой 4: Analytics Engine

**Компоненты:** `Statistical Module` | `ML Module (IF, LOF)` | `Fair Price Calculator`

### Statistical Module

Реализует базовые статистические методы как **Python-функции**, вызываемые AI-агентом через Executor:

- **IQR-метод** для детекции ценовых аномалий: значения вне `[Q1 - 1.5*IQR, Q3 + 1.5*IQR]` маркируются как выбросы
- **MAD (Median Absolute Deviation)** — робастная альтернатива стандартному отклонению, устойчива к выбросам
- **Percentile-based анализ** — определение, в какой перцентиль попадает цена конкретного лота относительно аналогов
- **Z-score** по средневзвешенной цене — для ответа на вопрос "отклонение > 30% от средневзвешенной" (прямое требование ТЗ)

### ML Module

- **Isolation Forest** — для многомерного обнаружения аномалий (учитывает одновременно цену, количество, регион, способ закупки). Обучается на всём историческом корпусе по каждому коду ЕНС ТРУ
- **LOF (Local Outlier Factor)** — для обнаружения локальных аномалий (закупка нормальная в целом, но аномальная для конкретного региона/ведомства)
- Модели **переобучаются** при каждом обновлении данных (инкрементально через partial_fit где возможно)

### Fair Price Calculator

Формализация метрики **Fair Price** — ключевое требование ТЗ:

```
FairPrice(enstru, kato, date) = MedianPrice(enstru, period) * RegionalCoeff(kato) * InflationIndex(date)
```

Где:

**1. Сравнительный подход (MedianPrice):**

- Берём все контракты с идентичным кодом ЕНС ТРУ за последние 4 квартала
- Используем **медиану** (а не среднее) — устойчивость к выбросам
- Минимальный размер выборки: N >= 5. При N < 5 расширяем до родительского кода ЕНС ТРУ или увеличиваем временное окно
- Формируем **доверительный интервал**: [Q1, Q3] — "нормальный" диапазон цен

**2. Региональный коэффициент (RegionalCoeff):**

```
RegionalCoeff(kato) = MedianPrice(enstru, kato) / MedianPrice(enstru, all_kato)
```

- Рассчитывается для каждого региона (уровень области по КАТО)
- Учитывает, что в Мангистауской области логистика дороже, чем в Алматы
- При отсутствии данных по региону — коэффициент = 1.0

**3. Временной фактор (InflationIndex):**

- Индекс потребительских цен (ИПЦ) от Бюро национальной статистики РК
- Приводим все исторические цены к текущему периоду: `AdjustedPrice = HistoricalPrice * (CPI_current / CPI_historical)`
- Для категорий с выраженной сезонностью (продукты питания, строительные материалы) — дополнительный сезонный коэффициент на основе исторических паттернов

**Итоговая оценка справедливости:**

| Отклонение от FairPrice | Вердикт |
|-------------------------|---------|
| +/- 15% | Нормально |
| 15–30% | Требует внимания |
| > 30% | Аномалия |

Каждая оценка сопровождается **метаданными**: размер выборки N, период, регион, метод расчёта, доверительный интервал.

### Почему так

- **Комбинация статистики + ML** — ТЗ допускает и рекомендует комбинированный подход. Статистика (IQR, MAD) — интерпретируема и быстра; ML (IF, LOF) — ловит нелинейные паттерны
- **Медиана вместо среднего** — госзакупки содержат экстремальные выбросы (ошибки ввода, сверхкрупные лоты), медиана к ним устойчива
- **Fair Price как формула** — делает метрику **воспроизводимой** (требование ТЗ: "объяснимый, воспроизводимый ответ"). Любой аналитик может пересчитать вручную
- **Региональный коэффициент** — прямое требование ТЗ ("корректировка цены в зависимости от места поставки")

## Слой 5: AI Agent Core (LLM)

**Компоненты:** `NLU Module` -> `Query Planner` -> `SQL/Py Executor` -> `Response Composer`

Это **центральный интеллектуальный слой** системы. Реализован как **ReAct-агент** (Reasoning + Acting) на базе LLM.

### NLU Module (Natural Language Understanding)

**Что делает:**

1. Принимает вопрос на казахском или русском языке
2. **Определяет язык** и при необходимости транслитерирует (KZ -> RU для единообразия обработки)
3. **Классифицирует тип вопроса** (prompt-based classification):
   - `search` — поиск конкретной закупки/организации
   - `comparison` — сравнение цен/объёмов между организациями/периодами
   - `analytics` — агрегированная статистика
   - `anomaly` — выявление аномалий
   - `fairness` — оценка справедливости цены/количества
4. **Извлекает сущности (NER):**
   - БИН организаций (12-значные числа)
   - Коды ЕНС ТРУ
   - Даты и периоды
   - Суммы и пороги ("отклонение > 30%")
   - Географию (города, регионы -> коды КАТО)
   - Номера лотов/объявлений

### Query Planner

**Что делает:**

- LLM получает **схему БД** (DDL таблиц + описание Materialized Views) и **набор доступных Python-функций** (statistical, ML, Fair Price) как tools
- На основе типа вопроса и извлечённых сущностей **планирует цепочку действий**

Пример для вопроса *"Найди закупки с отклонением цены > 30% от средневзвешенной по данному ENSTРУ"*:

```
Plan:
1. SQL: получить средневзвешенную цену из mv_price_statistics WHERE enstru_code = X
2. SQL: получить все лоты с enstru_code = X за указанный период
3. Python: рассчитать отклонение каждого лота от средневзвешенной
4. Python: отфильтровать лоты с отклонением > 30%
5. SQL: получить детали закупок (номера, заказчики, ссылки) для top-K результатов
6. Format: сформировать ответ по шаблону ТЗ
```

### SQL/Python Executor (Sandbox)

**Что делает:**

- Выполняет сгенерированные LLM-ом SQL-запросы и Python-код в **изолированной среде**
- SQL выполняется через **read-only** connection к PostgreSQL (защита от случайного DELETE/UPDATE)
- Python-код выполняется в **sandbox** (ограничение по CPU, памяти, запрет сетевых вызовов, импорт только разрешённых библиотек: numpy, scipy, sklearn)
- Результаты возвращаются в LLM для интерпретации

**Почему отдельный Executor, а не LLM считает сама:**

- Прямое требование ТЗ: *"LLM не выполняет математические расчеты сама, а использует внешние инструменты (Python/SQL)"*
- Гарантия точности: SQL-агрегаты математически точны, LLM-галлюцинации исключены для числовых данных
- Воспроизводимость: запросы логируются и могут быть перезапущены

### Response Composer

**Что делает:**

- Получает от Executor числовые результаты и форматирует ответ **строго по структуре ТЗ**:

```
1. Вердикт (1-3 предложения)
2. Параметры поиска (период, БИНы, фильтры)
3. Аналитика (сравнение, средние, отклонения)
4. Метрика оценки (какой метод использован)
5. Уверенность (качество данных, размер выборки, ограничения)
6. Детализация — Top-K закупок с ID и ссылками
```

- **Explainability** встроена в каждый ответ:
  - Ссылки на конкретные ID закупок и контрактов
  - Размер выборки N
  - Метод оценки (IQR / Isolation Forest / Fair Price formula)
  - Степень уверенности: `high` (N > 30), `medium` (10 <= N <= 30), `low` (N < 10)
  - Ограничения ("данные только по 5 организациям", "нет данных по этому региону за 2024")

### Реализация агента

```
Подход: ReAct Agent (LangChain / LlamaIndex)
LLM:    Claude / GPT-4 (API) или локальная модель для on-prem
Tools:
  - execute_sql(query) -> DataFrame
  - calculate_statistics(data, method) -> dict
  - detect_anomalies(data, model) -> list
  - calculate_fair_price(enstru, kato, date) -> dict
  - format_response(results, template) -> str
```

### Почему ReAct, а не RAG

- **RAG** (Retrieval-Augmented Generation) подразумевает поиск по документам. Наши данные — **структурированные таблицы**, а не тексты. RAG здесь неэффективен
- **ReAct** позволяет агенту **рассуждать пошагово** ("мне нужна средневзвешенная цена -> напишу SQL -> получу результат -> сравню с конкретным лотом -> сформирую вердикт") — это обеспечивает и точность, и explainability
- Элементы RAG используются **только** для: поиска по названиям лотов (нечёткий текстовый поиск через `pg_trgm` или встроенный поисковый индекс)

## Cross-cutting: Инфраструктура

### Airflow Scheduler

- Оркестрирует ETL-пайплайн: `ingest -> transform -> load -> refresh_materialized_views -> retrain_ml_models`
- Расписание: каждые 12 часов (запас для требования <= 24h)
- Алерты при сбоях (падение API, ошибки валидации)

### Secrets Vault

- Хранит API-токен goszakup.gov.kz (требование ТЗ: "защита токенов API")
- Хранит credentials к PostgreSQL
- Kubernetes Secrets или HashiCorp Vault

### Docker / Kubernetes

- Каждый компонент — отдельный Docker-контейнер
- `docker-compose.yml` для локальной разработки
- Helm Chart для Kubernetes deployment (on-prem requirement)
- Контейнеры: `ingestion`, `etl`, `postgres`, `analytics`, `agent`, `web`, `airflow-scheduler`, `airflow-worker`

## Соответствие требованиям ТЗ

| Требование ТЗ | Как реализовано |
|----------------|-----------------|
| Вопросы на KZ/RU | NLU Module с определением языка |
| Определение типа вопроса | Prompt-based classification в NLU |
| Точность фактов >= 85% | SQL-based extraction (не LLM-галлюцинации), Quality Gate на данных |
| Инкремент <= 24h | Journal Sync каждые 12h через Airflow |
| Аномалии (>30% отклонение) | Statistical Module (Z-score) + Isolation Forest |
| Справедливость цены | Fair Price Calculator (сравнительный + региональный + временной) |
| Завышение объёмов | mv_volume_trends (YoY comparison) + LOF |
| Explainability | Response Composer: ID, выборка, метод, уверенность |
| LLM не считает | SQL/Py Executor в sandbox, LLM только планирует |
| On-prem (Docker/K8s) | Полная контейнеризация, Helm Chart |
| Контроль качества | Quality Gate + data_quality_log |
| Защита токенов | Secrets Vault (K8s Secrets / HashiCorp Vault) |

## Риски и ограничения

1. **API goszakup.gov.kz нестабильность** — rate limiting, таймауты. Митигация: retry с backoff, кеширование raw data
2. **Малый размер выборки** — для редких кодов ЕНС ТРУ может быть N < 5 контрактов. Митигация: расширение до родительского кода, явное указание низкой уверенности
3. **Качество данных** — мусор в названиях, отсутствующие поля. Митигация: Quality Gate с логированием
4. **Стоимость LLM** — при использовании cloud API (Claude/GPT-4). Митигация: кеширование частых запросов, возможность on-prem LLM (Llama, Mistral)
5. **Инфляционные данные** — ИПЦ нужно загружать из внешнего источника (stat.gov.kz). Митигация: ежемесячное ручное/автоматическое обновление
